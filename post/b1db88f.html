草稿笔记

观察/趋势

  • 计算范畴：统计->规则（专家会议沉淀）->图->AI->复杂决策
  • 计算复杂度：静态线性工作流->静态图工作流->

论文，Hidden Technology Google

真正的 Machine Learning Code 只是很小的一部分。

问题

  • 学习/开发/调试的成本

    • 对接很多独立的引擎/框架
    • Ad-hoc/碎片化的集成/联调

  • 运行时质量

    • 破碎的上下游衔接（上面跑数据不确定，下面就可能接受到不对的结果，接口/时效性/一致性/故障恢复/中间状态管理（HDFS 存放到后面消费，但是不知道什么时候可以删除，存储爆掉））。
    • 盲目调度，存在很多优化的空间
    • 冗余/低效的数据传输/拷贝/计算等
    • 安全隐私

  • 运维效率

    • 多引擎/平台异构运维

胶水

已经有一些相关的工作，Workflow。
大家都希望在比较好的界面中通过 GUI 组合跑起来。

e.g. Azure ML Workflow， Google Kubeflow

  • 优点

    • 包容性好
    • 易用

  • 局限性

    • 组件质量/接口参差不齐->接口难做
    • 组件对于胶水来说是黑盒->跨接口优化难做

想法：从教水稻统一的分布式支撑。

目标

融合计算：本地编程、自动云化、渐进融合

核心挑战 1：统一的通用编程抽象/执行模型是什么

抽象就意味着，面临不同分布式计算模式的共同需求，混合模式下的需求。

核心挑战 2：如何解决通用和特定之间的固有矛盾

一般情况下，越通用，解决特定场景能力越弱。

融合计算编程抽象

为什么需要分布式计算？

以 MapReduce 为例，wordcount。

对于单机：存储空间不够、计算能力不构、

分布式计算目标：提供远超的单机的存储、计算能力，包装成单机的假象让上层没有意识到

状态 scale out

状态切分成 slice，不需要代码改动的方案：remote memory，file system。

需要代码改动的：KV，database

Shared state + duplicate 共享

想法

为了避免数据共享冲突，调整架构。

  • 冯诺依曼（控制流） -> 数据流
  • 需要改写代码

回到 MapReduce

拆分成两个任务，Map 任务，Reduce 任务。

表达能力：数据流二部图（Task Object）

但是，Master 是个 exception，Master 是有状态的，相对于 Master 和 Reduce。

无状态任务 vs 有状态任务

  • 无状态
    • 大多数数据流计算（MapReduce，Dryad，Spark）
    • 大多数 SOA 里面的 Method Call
    • 随便哪里都能跑
  • 有状态
    • 某些有迭代的数据流计算（Tensorflow，Grap Computation2）
    • 存储操作
    • 带状态的服务操作（大多数 Job Master）
    • 执行逻辑必须和状态绑在一起

编程抽象之 服务

服务 vs Actor

  • 服务 = [同构 Actor]*（一组同构 Actor 的集合），Actor 是单进程的一个
  • 尽量隐藏分布式计算带来的复杂度
    • Scale out
    • Elasticity 弹性，日夜差别
    • Load balance
    • Fail over

团队认为这样多线的是有相似性，优化能力的，所以服务更适合作为抽象单元。

数据流+服务，也可以 Map 映射到二部图中，这样在运行中有一个兜底的作用？？？
